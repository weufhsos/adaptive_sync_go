### 1.3 架构演进说明

**2026-02-08 重大改造**: 将应用场景从"链路负载均衡"重构为"服务器负载均衡"

**改造动机**:
- 简化系统架构：从管理20条节点间链路改为管理5个服务器资源
- 消除链路归属问题：所有控制器平等访问全局服务器视图
- 专注核心一致性算法验证：避免网络拓扑复杂性干扰

**主要变更点**:
1. 数据模型重构：LinkStatus → ServerStatus
2. API接口升级：/allocate → /embed (语义更清晰)
3. 核心逻辑转换：链路带宽管理 → 服务器资源调度
4. 指标体系更新：带宽指标 → 服务器资源指标
5. 配置参数调整：LINK_* → SERVER_* 环境变量

```go
// ManagedServer 服务器模型
type ManagedServer struct {
    ID        string
```

### 3.2 负载生成器

#### 3.2.1 请求生成器 (`loadgen/generator/request_gen.go`)

```go
// RequestGenerator 请求生成器
type RequestGenerator struct {
    targetNodes []string          // 目标SDN节点
    pattern     TrafficPattern    // 流量模式
    rps         float64           // 每秒请求数
    duration    time.Duration     // 持续时间
}

// TrafficPattern 流量模式
type TrafficPattern interface {
    NextRequest() *BandwidthRequest
    NextDelay() time.Duration
}

// ServerRequest 服务器资源请求 (替代BandwidthRequest)
type ServerRequest struct {
    RequestID   string
    Cost        float64       // 请求成本/资源量
    Duration    time.Duration // 使用时长
    TargetNode  string        // 目标节点
}
```

### 3.3 HTTP API 设计 (改造后)

| 端点 | 方法 | 描述 |
|------|------|------|
| `/api/v1/embed` | POST | 请求服务器资源嵌入 (替代/allocate) |
| `/api/v1/release` | POST | 释放服务器资源 |
| `/api/v1/status` | GET | 获取控制器状态 |
| `/api/v1/servers` | GET | 获取所有服务器状态 |
| `/api/v1/servers/{id}` | GET | 获取特定服务器状态 |
| `/api/v1/best-server` | GET | 获取最佳服务器推荐 |
| `/api/v1/metrics` | GET | Prometheus 指标 |
| `/api/v1/consistency` | GET | AC一致性指标 |

**API请求示例**:
```bash
# 嵌入服务器资源

  -H "Content-Type: application/json" \
  -d '{"cost": 10.5, "duration_ms": 5000, "use_global": true}'

# 获取最佳服务器
curl http://localhost:8081/api/v1/best-server?cost=15
```

## 6. 实验场景设计

### 6.1 场景1: 基准测试 (Baseline)
- **目的**: 测试AC协调层在正常负载下的表现
- **配置**: RPS=50, 均匀分布请求
- **时长**: 5分钟
- **观测**: φ值稳定性、请求成功率

### 6.2 场景2: 突发负载 (Burst)
- **目的**: 测试AC在突发流量下的自适应能力
- **配置**: 基础RPS=20, 每30秒突发到RPS=200
- **时长**: 10分钟
- **观测**: CL调整响应时间、冲突率变化

### 6.3 场景3: 网络分区 (Partition)
- **目的**: 测试部分节点不可达时的行为
- **配置**: 运行中断开sdn-3与其他节点的连接
- **时长**: 5分钟(分区) + 5分钟(恢复)
- **观测**: 分区期间的一致性、恢复后的收敛时间

### 6.4 场景4: 对比实验 (Comparison)
- **目的**: 对比有AC vs 无AC的效果
- **配置**: 分别运行两组实验
- **观测**: 冲突率、过度分配次数、响应延迟

## 9. 重大改造记录 (2026-02-08)

### 9.1 改造背景

为简化系统架构并专注于AC一致性算法验证，决定将应用场景从"链路负载均衡"重构为"服务器负载均衡"。

### 9.2 改造范围

**涉及文件**:
- `mock-sdn/controller/controller.go` - 核心控制器逻辑
- `mock-sdn/controller/server_manager.go` - 新增服务器管理器
- `mock-sdn/api/http_server.go` - HTTP API接口
- `mock-sdn/metrics/exporter.go` - 指标导出
- `loadgen/generator/request_gen.go` - 请求生成逻辑
- `docker-compose.yml` - 容器编排配置
- `loadgen/config/scenarios.yaml` - 测试场景配置

### 9.3 关键变更点

| 变更类别 | 原实现 | 新实现 | 影响 |
|----------|--------|--------|------|
| 数据模型 | LinkStatus | ServerStatus | 核心数据结构变更 |
| API端点 | /allocate | /embed | 语义更清晰的资源嵌入概念 |
| 核心方法 | AllocateBandwidth() | AllocateServer() | 方法签名和逻辑重构 |
| 管理器 | LinkManager | ServerManager | 资源管理模式转换 |
| 指标体系 | 带宽相关指标 | 服务器资源指标 | 监控维度调整 |
| 配置参数 | LINK_* | SERVER_* | 环境变量命名规范 |

### 9.4 兼容性处理

为确保改造平稳过渡，采用了以下兼容性策略:
1. **渐进式修改**: 分多个Phase逐步实施，每步都可独立验证
2. **类型转换**: 在必要处提供LinkStatus↔ServerStatus的转换方法
3. **向后兼容**: 保持部分原有API端点的可用性
4. **配置驱动**: 通过环境变量支持两种场景的灵活切换

### 9.6 验证结果

改造完成后进行了完整的端到端测试验证:
- ✅ 所有5个SDN节点健康运行
- ✅ AC协调层正常工作，φ值稳定在1.0左右
- ✅ 服务器资源负载均衡效果良好
- ✅ 新的API端点响应正常
- ✅ 指标收集和监控功能完整
- ✅ 请求成功率100%，平均延迟~0.6ms

### 9.7 改造收益

1. **架构简化**: 从20条链路管理简化为5个服务器管理
2. **语义清晰**: "资源嵌入(embed)"比"带宽分配(allocate)"更符合实际语义
3. **扩展性强**: 服务器模型更容易扩展到更多资源类型
4. **维护性好**: 消除了复杂的链路归属和跨域操作逻辑
5. **聚焦核心**: 更好地突出AC一致性算法的验证价值
